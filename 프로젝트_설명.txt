 1. 도메인(Domain)이란?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  앱에게 "이 문서가 어떤 업종 것인지" 미리 알려주는 설정

  같은 단어도 업종마다 의미가 다릅니다.
  예) "재고" → 뷰티에서는 화장품 SKU 단위 / 공급망에서는 부품 수량

  도메인을 설정하면:
  → AI가 해당 업종 언어로 문서를 이해
  → 분석 포커스가 자동으로 맞춰짐
  → 채팅 예시 질문도 업종에 맞게 바뀜

  ---

  지원 도메인 목록:

    뷰티/이커머스   →  SKU, MOQ, 리드타임, 채널, D2C
    공급망/재고     →  재발주점, 안전재고, 납기, A/B/C 등급
    에너지          →  발전량, 소비량, 역률, 피크전력
    제조/생산       →  OEE, 가동률, 불량률, 다운타임
    물류            →  배송성공률, 허브, 노선, 차량
    금융            →  신용등급, 이상거래, 자산, 리스크

  ---

  도메인은 어떻게 만들었나?

    domains/ 폴더 안에 도메인마다 파이썬 파일 하나씩 작성

    예) domains/beauty.py:

        PRESET = {
            "terminology": ["SKU", "MOQ", "리드타임", ...],
            "entity_types": {"product": 분홍, "supplier": 보라, ...},
            "theme_color": "#E91E63",
            "app_icon": "🛒",
        }

    처음 보는 도메인이 입력되면 Claude API를 한 번 호출해서
    위 구조를 자동으로 생성함 (코드 없이 새 도메인 추가 가능)

 2. FK(Foreign Key, 외래키)란?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  여러 테이블을 이어주는 공통 열쇠

  엑셀로 비유하면:
  → 시트1(제품목록)과 시트2(판매실적)을 VLOOKUP으로 연결하는 것

  ---

  실제 예시 (뷰티 도메인):

  ┌─────────────────────────────────────────────────────┐
  │ MST_PRODUCT (제품 마스터)                           │
  │                                                     │
  │  PRODUCT_ID │ PRODUCT_NAME         │ SUPPLIER_ID   │
  │  ───────────┼──────────────────────┼─────────────  │
  │  PRD001     │ 클린잇제로 클렌징밤   │ SUP001        │
  │  PRD002     │ 선프로텍터 선크림     │ SUP004        │
  └──────┬──────────────────────────────────────────────┘
         │  PRODUCT_ID 가 일치 (FK)
         ↓
  ┌─────────────────────────────────────────────────────┐
  │ FACT_MONTHLY_SALES (월별 판매 실적)                 │
  │                                                     │
  │  PRODUCT_ID │ YEAR_MONTH │ SALES_QTY │ SALES_AMOUNT │
  │  ───────────┼────────────┼───────────┼──────────── │
  │  PRD001     │ 2024-01    │ 370개     │ 8,140,000원 │
  │  PRD002     │ 2024-01    │ 290개     │ 5,220,000원 │
  └─────────────────────────────────────────────────────┘

  이 앱은 컬럼명 끝이 _ID, _CD, _NO 인 열을 보고
  FK 관계를 자동으로 감지합니다.
  SQL JOIN 코드 없이 테이블 간 연결을 파악합니다.

  ---

  감지 결과 예시:

    FACT_MONTHLY_SALES.PRODUCT_ID  →  MST_PRODUCT
    FACT_INVENTORY.PRODUCT_ID      →  MST_PRODUCT
    FACT_INVENTORY.LOCATION_ID     →  MST_LOCATION
    MST_PRODUCT.SUPPLIER_ID        →  MST_SUPPLIER
    MST_SUPPLY_ROUTE.OFFICE_CD     →  MST_OFFICE

 3. 지식그래프(Knowledge Graph)란?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  개념(노드)과 연결(엣지)으로 이루어진 관계 지도

  왜 필요한가?
  → 단순 검색은 "A 문서 어디에 클렌징밤이 언급됐는지"만 찾음
  → 지식그래프는 "클렌징밤이 어느 공급업체에서 오고,
     어느 창고에 있고, 지난달 얼마나 팔렸는지"까지 한번에 추적

  ---

  실제 예시 (뷰티 도메인):

  [MST_PRODUCT]
      PRD001 "클린잇제로 클렌징밤"
           │
           ├─ SUPPLIER_ID ──→ [MST_SUPPLIER] SUP001 "코스메랩"
           │                       리드타임: 14일, MOQ: 100
           │
           ├─ PRODUCT_ID ───→ [FACT_MONTHLY_SALES]
           │                       2024-01: 370개 / 8,140,000원
           │                       2024-02: 324개 / 7,128,000원
           │
           └─ PRODUCT_ID ───→ [FACT_INVENTORY]
                                   재고: 346개 / 안전재고: 394개
                                   커버리지: 2.1주 ← 위험!

  이 그래프를 통해:
  "PRD001 재고가 부족하면 → 공급업체 SUP001에 발주 → 리드타임 14일"
  이런 연쇄 관계를 AI가 파악해서 답변에 활용합니다.

  ---

  기술 스택:
  - NetworkX   : 파이썬에서 그래프 구조 구성 (노드/엣지 관리)
  - PyVis      : 인터랙티브 HTML 시각화 (화면에 보이는 그래프)
  - Claude API : 문서에서 엔티티(개념)와 관계 자동 추출

 4. RAG(Retrieval-Augmented Generation)란?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  문서를 잘게 쪼개 저장해두고
  질문이 오면 관련 부분만 찾아서 AI에게 전달하는 방식

  왜 필요한가?
  → AI는 한번에 처리할 수 있는 텍스트 양에 한계가 있음
  → 수백 페이지 문서를 통째로 넣으면 너무 느리고 비쌈
  → 관련 부분만 골라주면 빠르고 정확

  ---

  동작 순서:

  ① 청킹 (파일을 조각냄)

      MST_PRODUCT.csv 전체 내용
      ──────────────────────────
      "PRD001, 클린잇제로 클렌징밤, 스킨케어, 22000원, SUP001..."
      "PRD002, 선프로텍터 선크림, 스킨케어, 18000원, SUP004..."
      ...
                  ↓
      [청크 1]  PRD001~PRD003 내용  (약 300단어)
      [청크 2]  PRD004~PRD007 내용  (약 300단어)
      [청크 3]  PRD008~PRD012 내용  (약 300단어)

  ② 벡터화 (숫자 배열로 변환)

      "클린잇제로 클렌징밤, 스킨케어, 22000원"
                  ↓  (AI 언어모델로 의미 추출)
      [0.23, -0.41, 0.87, 0.12, -0.65, ...] (768개 숫자)

      → 의미가 비슷한 문장은 비슷한 숫자 배열이 됨

  ③ 저장 (ChromaDB)

      청크 1 → [벡터]  저장
      청크 2 → [벡터]  저장
      청크 3 → [벡터]  저장
      ...

  ④ 검색 (질문이 들어오면)

      "재고 위험 제품 알려줘"
                  ↓  (질문도 벡터로 변환)
      [0.21, -0.39, 0.91, ...]

      → 저장된 벡터들과 유사도 계산
      → 가장 유사한 청크 TOP 3 반환

      결과:
        [청크 A]  FACT_INVENTORY - PRD001 재고 346개, 안전재고 394개 ← 미달!
        [청크 B]  FACT_INVENTORY - PRD002 재고 309개, 안전재고 384개 ← 미달!
        [청크 C]  FACT_STOCKOUT_EVENT - 2024-03 품절 발생 이력

  ⑤ AI 답변 생성

      위 3개 청크를 Claude에게 전달
      → "현재 재고 위험 제품: 클린잇제로 클렌징밤(PRD001)과
         선프로텍터 선크림(PRD002)이 안전재고 미달입니다..."

  ---

  기술 스택:
  - ChromaDB   : 벡터 데이터베이스 (청크 저장 및 검색)
  - paraphrase-multilingual-MiniLM-L12-v2
               : 한국어 지원 임베딩 모델 (텍스트 → 벡터 변환)

 5. 실제 데이터 예시로 보는 전체 흐름
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  시나리오: 뷰티 브랜드 담당자가 앱을 처음 사용하는 상황

  ---

  STEP 1. 도메인 선택

    → "뷰티 / 이커머스" 클릭
    → 앱이 뷰티 용어 세트 로드
       (SKU, MOQ, 리드타임, 채널, CPNP 등)

  ---

  STEP 2. 파일 업로드

    → MST_PRODUCT.csv 업로드
    → FACT_MONTHLY_SALES.csv 업로드
    → FACT_INVENTORY.csv 업로드

  ---

  STEP 3-A. FK 자동 감지

    앱이 컬럼명을 분석:
    - FACT_MONTHLY_SALES.PRODUCT_ID   → MST_PRODUCT 연결 감지
    - FACT_INVENTORY.PRODUCT_ID       → MST_PRODUCT 연결 감지
    - FACT_INVENTORY.LOCATION_ID      → MST_LOCATION 연결 감지
    - MST_PRODUCT.SUPPLIER_ID         → MST_SUPPLIER 연결 감지

  ---

  STEP 3-B. 지식그래프 구축

    감지된 FK 연결을 그래프로 시각화:

      [PRD001 클린잇제로] ─SUPPLIER_ID─→ [SUP001 코스메랩]
      [PRD001 클린잇제로] ─PRODUCT_ID──→ [판매: 370개/월]
      [PRD001 클린잇제로] ─PRODUCT_ID──→ [재고: 346개, 위험]

    화면에 인터랙티브 그래프로 표시
    → 노드 클릭하면 연결된 관련 정보 드릴다운 가능

  ---

  STEP 3-C. RAG 인덱싱

    업로드된 파일 내용을 300단어씩 청킹
    → 각 청크를 벡터로 변환 → ChromaDB에 저장
    완료 시: "총 47개 청크 인덱싱 완료" 메시지

  ---

  STEP 4. AI 자동 분석 (탭으로 제공)

    [핵심 요약 탭]
    → "3개 제품 재고 위험, 선크림은 여름 시즌 피크 전 발주 필요"

    [액션 아이템 탭]
    → "오늘: PRD002 긴급 발주 / 이번 주: 안전재고 기준 재검토"

    [원인 분석 탭]
    → 5-Why 방식으로 재고 부족 근본 원인 추적

    [보고서 탭]
    → 임원 보고용 운영 현황 보고서 자동 초안 생성

  ---

  STEP 5. 채팅

    사용자 입력:
    "선프로텍터 선크림 지금 재고 괜찮아?"

    앱 내부 동작:
    1) "선크림 재고" 키워드로 RAG 검색
       → FACT_INVENTORY에서 PRD002 관련 청크 추출
          재고 309개, 안전재고 384개, 커버리지 2.1주

    2) 지식그래프에서 PRD002 연결 탐색
       → SEASONAL_PEAK: 여름
       → 공급업체 SUP004, 리드타임 7일

    3) Claude에게 전달 → 답변 생성

    Claude 답변:
    "선프로텍터 선크림(PRD002) 현재 재고는 309개로
